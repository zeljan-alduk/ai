[2026-04-24 22:35 UTC] backend-engineer@meridian-labs — wave 3C: meridian-fs MCP server
Built mcp-servers/meridian-fs/ (@meridian/mcp-fs): first-party stdio MCP
server backed by @modelcontextprotocol/sdk 1.29, zod 3.25, vitest 2.1.
Five tools: fs.read / fs.write / fs.list / fs.stat / fs.search; fs.delete
and fs.move flagged TODO(v1). ACL is the security core — three layers:
lexical containment after path.resolve, per-component lstat/realpath check
to refuse symlinks pointing outside roots, final realpath check on the
target (or its deepest existing ancestor for new-file writes). Roots
declared via --roots / MERIDIAN_FS_ROOTS / --config json; spec syntax
deviated to comma-separated <abs-path>:<ro|rw> pairs (brief said colon-
separated but that breaks Windows drive letters; documented). 32 tests
across 3 files all green: traversal blocked, symlink escapes blocked,
ro-root writes refused, in-process MCP client round-trips fs.read /
fs.write via InMemoryTransport. Errors are structured {code, message}
with codes NOT_FOUND | PERMISSION_DENIED | OUT_OF_BOUNDS | TOO_LARGE |
INTERNAL. Caps: read 4MiB, write 8MiB, search 200 hits / 5000 lines /
1000 files. Least-confident security check flagged in README for the
auditor: TOCTOU between ACL realpath and the actual read/write syscall.
Mitigation candidate: openat with O_NOFOLLOW per component plus fstat
inode re-check. Touched only mcp-servers/meridian-fs/.

[2026-04-25 15:16 UTC] orchestrator — wave 7 follow-up: rowCount fix (recovery #8)
Smoke DELETE bug from the wave-7 LIVE entry: traced to the Neon
adapter (platform/storage/src/pool.ts) calling sql.query with
`fullResults: false`, which makes @neondatabase/serverless return
only the rows array. For DELETE/UPDATE without RETURNING the array
is empty, and the adapter's `rowCount: rows.length` was always 0.

in case a future Neon version omits the field on certain commands.

mutation queries (audit-replay logic in the registry, eval promotion
gates, etc.).

`@aldo-ai/cache` shipped at platform/cache/. Helicone-style
LLM-response cache layer that intercepts at the gateway, not the
engine: identical request -> replay prior response -> save the
provider call + cost.

Tables (migration 017_llm_response_cache.sql):
  - llm_response_cache (tenant_id, key) PK; JSONB response + usage;
    NUMERIC(10,6) cost_saved_usd; hit_count INT; created_at +
    last_hit_at + expires_at TIMESTAMPTZ. Indexes on
    (tenant_id, expires_at) and (model). FK to tenants.
  - tenant_cache_policy (tenant_id PK FK -> tenants); enabled
    BOOL DEFAULT TRUE; ttl_seconds INT DEFAULT 86400;
    cache_sensitive BOOL DEFAULT FALSE; updated_at TIMESTAMPTZ.

Cache-key inputs (SHA-256 over canonical-JSON, stable-stringified):
  - model id (opaque string),
  - privacy_tier  <- CRITICAL: a `sensitive` request can NEVER read
                     a `public` row of the same prompt; the key
                     namespaces them apart. Tier-leak guard,
                     CLAUDE.md non-negotiable #3.
  - messages array (preserves order; system prompt is just a
    role='system' message, hashed identically),
  - tools schema array,
  - temperature, top_p (extension), max_tokens,
  - response_format, stop sequences, seed.

Excluded from key (deliberately): runId / traceId / message id /
abort signal — they vary per call but don't change the response.

GatewayMiddleware integration:
  - `CacheMiddleware` implements GatewayMiddleware (mirrored locally
    so the cache package doesn't take a hard dep on @aldo-ai/gateway).
    `before` computes the key + stamps it on a per-CallContext
    WeakMap. `after` captures every Delta; on the `end` frame it
    persists the assembled response + usage to the cache (best-effort,
    fire-and-forget).
  - `wrapGatewayWithCache(gw, opts)` wraps a ModelGateway so the
    READ side fires too: hits short-circuit and replay the captured
    deltas WITHOUT calling the inner gateway; misses delegate (and
    let the after-hook persist on the way out). This is the surface
    that actually saves API call cost — the middleware alone only
    fills the cache; the wrapper reads from it.
  - Streaming replay choice (documented in middleware.ts header):
    cached responses replay as a single full-text Delta + every
    captured tool_call + a synthetic `end` frame. Provider chunk
    boundaries are NOT reconstructed — they're a function of network
    behaviour, not model output. Same trade Helicone makes.

Privacy:
  - Default policy: enabled + 24h TTL + sensitive-tier SKIPS the
    cache. Documented in policy.ts header as a deliberate safety
    choice; opt-in (`cacheSensitive=true`) is wired but the wave-17
    UI surface for owner-only audited toggle is the follow-up.
