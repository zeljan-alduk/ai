## 1. Architecture snapshot

ALDO AI runs on a single VPS (`vps-77bcd56d`, see STATUS.md) under
Docker Compose. Three containers — `aldo-postgres` (Postgres 16, named
volume), `aldo-api` (Hono, internal port 8081), `aldo-web` (Next.js,
internal port 8082) — sit behind an edge nginx. The edge is a
docker-managed nginx already running on the host, so ai.aldo.tech
ships as an additional vhost (`zzz-ai-aldo-tech.conf`) in that
container's mounted `conf.d/` rather than its own host nginx. A
separate systemd service, `aldo-deploy-webhook`, runs a Python stdlib
HTTP server on `127.0.0.1:9999` (or `0.0.0.0:9999` when the edge is
external); nginx proxies `/_admin/*` to it. Pushes to
`main` or `claude/*` trigger `.github/workflows/deploy-vps.yml`, which
POSTs to `https://ai.aldo.tech/_admin/deploy` with a bearer token and
the webhook runs `scripts/vps-deploy.sh`.

Paths on the host:

| Path | Contents |
|---|---|
| `/opt/aldo-ai/repo` | Git checkout (the deploy script does `git reset --hard FETCH_HEAD` here) |
| `/opt/aldo-ai/data` | Reserved for host-side data; Postgres uses a named docker volume |
| `/opt/aldo-ai/secrets/` | `postgres_password`, `jwt_secret`, `secrets_master_key`, `deploy_token` (chmod 600) |
| `/opt/aldo-ai/logs/deploy.log` | Per-deploy stdout from `vps-deploy.sh` |
| `/opt/aldo-ai/logs/webhook.log` | Webhook stdout (also via `journalctl -u aldo-deploy-webhook`) |
| `/opt/aldo-ai/docker-compose.yml` | Generated by `scripts/vps-emit-compose.sh` |

## 2. Deploy and rollback

### 2.1 Normal deploy

CI deploys automatically on push to `main` or
`claude/ai-agent-orchestrator-hAmzy`. `paths-ignore` skips deploys for
docs-only changes (`docs/**`, `**.md`, `CLAUDE.md`,
`DEVELOPMENT_LOG.txt`).

Manual dispatch from a workstation:

```sh
gh workflow run deploy-vps.yml -f branch=claude/ai-agent-orchestrator-hAmzy
gh run watch  # follow the run; smoke-test step at the end exits non-zero on failure
```

Manual dispatch by hand against the webhook (token from
`/opt/aldo-ai/secrets/deploy_token` on the VPS, or the
`VPS_DEPLOY_TOKEN` GitHub secret):

```sh
TOKEN='...'  # bearer
curl -sS --fail-with-body --max-time 600 \
  -X POST https://ai.aldo.tech/_admin/deploy \
  -H "Authorization: Bearer ${TOKEN}" \
  -H 'Content-Type: application/json' \
  -d '{"branch":"main"}'
```

The webhook serialises deploys in-process; a concurrent POST returns
`409 deploy_in_progress`. The script regenerates `docker-compose.yml`
from the pulled source, runs `docker compose up -d --build`, then polls
`http://127.0.0.1:8081/health` for up to two minutes. Non-zero exit
means the API never came back healthy; the webhook returns 500 and the
GitHub Action fails.

### 2.2 Rollback

There is no separate rollback artefact. Rollback = redeploy a known-good
commit. Pick a SHA from `git log --oneline` (or the last green
`deploy-vps` run on GitHub), then either:

**Option A — via webhook from a workstation** (preferred, leaves an
audit trail in the deploy webhook log and the GH Actions run):

```sh
# 1. Find the last green SHA.
gh run list --workflow deploy-vps.yml --limit 10
# 2. Reset the deploy branch to that SHA on the remote.
git fetch origin
git checkout claude/ai-agent-orchestrator-hAmzy
git reset --hard <SHA>
git push --force-with-lease origin claude/ai-agent-orchestrator-hAmzy
# Push fires deploy-vps.yml automatically.
```

`--force-with-lease` is allowed on the deploy branch because the only
"truth" we care about is what is running on the VPS. Do **not** force
push to `main` for a rollback — open a revert PR instead.

**Option B — directly on the VPS** when GitHub is unavailable or you
need to roll back faster than CI can run:

```sh
ssh root@<vps>
cd /opt/aldo-ai/repo
git fetch --depth 50 origin
git reset --hard <SHA>
sudo bash /opt/aldo-ai/repo/scripts/vps-deploy.sh claude/ai-agent-orchestrator-hAmzy
# The script regenerates compose, rebuilds, waits for /health.
```

Option B leaves the local checkout ahead of `origin`. Push the SHA up
afterwards (Option A workflow) so CI and the host agree.

### 2.3 What the deploy script will not do

- It will not run database migrations on its own. Migrations run on API
  boot inside the `aldo-api` container, so a deploy that only changes
  application code is enough to migrate.
- It will not preserve uncommitted local changes on the VPS — the first
  step is `git reset --hard FETCH_HEAD` followed by `git clean -fd`.
- It will not redeploy if the requested branch is not on the webhook
  allowlist (`main` or `claude/*`).

## 3. Common 5xx triage

Public health probes:

- `https://ai.aldo.tech/health` — should return `200` with `{"ok":true,...}`.
- `https://ai.aldo.tech/_admin/health` — webhook liveness, no auth.

### 502 / 504 from the edge

The edge nginx is up but the upstream container is not. SSH in:

```sh
ssh root@<vps>
docker compose -f /opt/aldo-ai/docker-compose.yml ps
```

If `aldo-api` or `aldo-web` is `exited` or `restarting`, tail logs:

```sh
docker compose -f /opt/aldo-ai/docker-compose.yml logs --tail=200 aldo-api
docker compose -f /opt/aldo-ai/docker-compose.yml logs --tail=200 aldo-web
```

Most common cause is a container that crashed on boot — typically a bad
migration, missing env var, or an LLM provider env variable change that
broke startup validation. Fix the root cause (or roll back per §2.2) and
`docker compose up -d` to bring it back.

### 401 unexpected on authenticated routes

Symptom: pages that worked yesterday redirect to `/login`, or
`/v1/auth/me` returns 401 for a session that should be valid.

This is the auth-proxy / cookie family. The web app stores sessions in
an HTTP-only cookie (`aldo_session`). Server components read it via
`lib/session.ts`; client mutations go through the
`/api/auth-proxy/[...path]` route handler which unwraps the cookie
server-side and re-issues with `Authorization: Bearer …`. Two failure
modes are documented in `DEVELOPMENT_LOG.txt`:

1. **Cookie mismatch after a JWT secret rotation.** If
   `/opt/aldo-ai/secrets/jwt_secret` changed, every existing session is
   stale; clients see 401 from `/v1/auth/me`. Resolution is no-op —
   users log in again. If you did not intentionally rotate, check the
   secret file's mtime.
2. **Auth-proxy bypass regression.** A regression test
   (`apps/web-e2e/tests/post-signup.spec.ts`) covers this. Failure mode:
   middleware allows a path through that should require auth, then the
   API rejects with 401. Roll back to last green and bisect.

### 500 from `/v1/*`

Almost always either a transient LLM upstream error or a migration that
landed half-applied. Tail API logs:

```sh
docker compose -f /opt/aldo-ai/docker-compose.yml logs --tail=200 aldo-api | grep -E '500|ERROR|panic|stack'
```

LLM upstream failures (Anthropic / OpenAI / Google rate limits, a model
hard-deprecation, a 5xx from the provider) are the most common cause.
The model gateway has a fallback chain per agent capability class, so a
single provider failure should *not* surface as a 500 — if it does,
check that the failing call's agent has fallbacks declared and that the
gateway routed through them. If not, the gateway itself is the bug;
file an issue and roll back the most recent platform change to
`platform/engine/` or `platform/router/`.

### Edge nginx itself

The edge proxy is a docker-managed nginx container, not host nginx.
To reload after a vhost change:

```sh
docker exec <edge-container> nginx -t   # validate
docker exec <edge-container> nginx -s reload
```

Find the container name with `docker ps --format '{{.Names}}'` (it owns
0.0.0.0:80 and 0.0.0.0:443).

## 4. Database backup and restore

There is **no scheduled backup job today.** This is a known gap; the
runbook below is the manual procedure.

### 4.1 Ad-hoc backup

```sh
ssh root@<vps>
docker exec -t $(docker ps -qf name=aldo-postgres) \
  pg_dump -U aldo -d aldo --format=custom --file=/tmp/aldo.dump
docker cp $(docker ps -qf name=aldo-postgres):/tmp/aldo.dump ./aldo-$(date +%Y%m%d-%H%M%S).dump
docker exec $(docker ps -qf name=aldo-postgres) rm /tmp/aldo.dump
# Move the dump off the host.
scp ./aldo-*.dump operator@<offsite>:/backups/
```

Substitute the actual DB user / database name if the compose file has
diverged; check with `docker exec <pg> env | grep POSTGRES_`.

### 4.2 Restore from a dump

```sh
# On the VPS, with a dump file at /tmp/aldo.dump:
docker compose -f /opt/aldo-ai/docker-compose.yml stop aldo-api aldo-web
docker exec -i $(docker ps -qf name=aldo-postgres) \
  pg_restore -U aldo -d aldo --clean --if-exists < /tmp/aldo.dump
docker compose -f /opt/aldo-ai/docker-compose.yml start aldo-api aldo-web
```

Stopping the application containers first prevents writes mid-restore.
Smoke-test `/health` afterwards.

### 4.3 Named volume snapshot (filesystem-level)

For a faster point-in-time backup that captures the entire data dir
including WAL state:

```sh
docker run --rm \
  -v aldo-ai_postgres_data:/var/lib/postgresql/data:ro \
  -v $(pwd):/backup \
  alpine tar czf /backup/pgdata-$(date +%Y%m%d-%H%M%S).tar.gz -C /var/lib/postgresql/data .
```

Volume name may differ; verify with `docker volume ls | grep postgres`.

## 5. Restart commands

Use the smallest hammer that fixes the problem.

| Symptom | Command | When |
|---|---|---|
| API returning 500s on every request, logs show stuck connection pool | `docker compose -f /opt/aldo-ai/docker-compose.yml restart aldo-api` | Postgres is fine, API needs to drop and re-open pools |
| Web bundle stale after a deploy that didn't actually swap | `docker compose -f /opt/aldo-ai/docker-compose.yml restart aldo-web` | Rare; `up -d --build` should handle this |
| Postgres OOM or stuck query | `docker compose -f /opt/aldo-ai/docker-compose.yml restart aldo-postgres` then restart `aldo-api` | Last resort — closes every active session |
| Webhook unresponsive (`/_admin/health` times out) | `systemctl restart aldo-deploy-webhook` | When CI says "webhook unreachable" |
| Edge proxy serving stale config | `docker exec <edge> nginx -t && docker exec <edge> nginx -s reload` | After hand-editing a vhost in `conf.d/` |
| Everything | `docker compose -f /opt/aldo-ai/docker-compose.yml down && docker compose -f /opt/aldo-ai/docker-compose.yml up -d` | Full bounce; ~30s of downtime |

## 6. Logs — where they live, how to tail

| Source | Path / command |
|---|---|
| API stdout | `docker compose -f /opt/aldo-ai/docker-compose.yml logs -f aldo-api` |
| Web stdout | `docker compose -f /opt/aldo-ai/docker-compose.yml logs -f aldo-web` |
| Postgres | `docker compose -f /opt/aldo-ai/docker-compose.yml logs -f aldo-postgres` |
| Deploy script | `tail -f /opt/aldo-ai/logs/deploy.log` |
| Webhook (Python) | `tail -f /opt/aldo-ai/logs/webhook.log` or `journalctl -u aldo-deploy-webhook -f` |
| Edge nginx access / error | Inside the edge container: `docker exec <edge> tail -f /var/log/nginx/access.log /var/log/nginx/error.log` (TBD: confirm exact path on the edge container — depends on its baked image) |

Log rotation: docker's default `json-file` driver does not rotate. If a
container goes chatty during an incident, free disk with
`docker system prune -f` (cautious — this also removes stopped
containers and dangling images).

## 8. When you must wake the founder

The decision tree for "do I escalate, or can this wait until business
hours?" — phrased for whoever inherits the pager.

```
START
│
├── Is ai.aldo.tech returning 5xx for >5 minutes on the public health probe?
│       │
│       ├── YES → SEV1 (full outage). Wake the founder immediately.
│       │        Begin §3 triage. Post to /status within 30 minutes.
│       │
│       └── NO → continue.
│
├── Is a paying customer's tenant unable to log in, run an agent, or
│   read their runs (for >15 minutes)?
│       │
│       ├── YES → SEV2 (major degradation). Wake the founder unless
│       │        you can identify and fix within 30 minutes (e.g. a
│       │        rollback per §2.2). If you start a rollback and it
│       │        doesn't bring the customer back inside 30 minutes,
│       │        escalate then.
│       │
│       └── NO → continue.
│
├── Is the deploy pipeline broken (no green deploy in >2 hours,
│   webhook returning 500 / 409 indefinitely)?
│       │
│       ├── YES → SEV3 (blocks engineering, not customers). Do NOT
│       │        wake the founder. File a GitHub issue, write up the
│       │        symptom, address in business hours.
│       │
│       └── NO → continue.
│
└── Did you discover something that looks like a security incident
    (credential leak, suspicious /_admin/* traffic in webhook.log,
    a delete-everything API call, an unauthorised tenant access)?
            │
            ├── YES → SEV1 regardless of customer impact. Wake the
            │        founder. Do NOT take destructive action (don't
            │        rotate secrets, don't delete logs) until the
            │        founder is on the call — preserving the audit
            │        trail matters more than reacting fast.
            │
            └── NO → file an issue, address in business hours.
```

A SEV1 wake-up is "phone call until they answer" + email +
`info@aldo.tech` thread. SEV2 is one call attempt + email; if no answer
in 15 minutes, decide on your own whether to roll back (default: yes,
roll back to last green). SEV3 never wakes anyone.

After every SEV1 / SEV2: write a postmortem within 72 hours and append
a one-line entry to `DEVELOPMENT_LOG.txt`.
