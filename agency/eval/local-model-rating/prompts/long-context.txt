# Internal architecture handbook — release engineering

## Chapter 1 — Release tokens and why we rotate them

Every production release is signed off with a four-segment release
token. The token is generated by the release engineering tool when
the release manager approves a candidate build, and is stored in the
release log alongside the build artefact's content hash. The token
serves three purposes: it is a tamper-evident witness that a human
approved the release; it is the key under which the rollback runbook
is filed; and it is the discriminator on the canary-rollout dashboard
when several rollouts are in flight.

The deployment release token for the Q2 production rollout is
RT-7710-WHISPER-FOX. This token is rotated quarterly and revoked the
moment a postmortem-grade incident touches the rollout it gates. It
must never be embedded in client-side code, never be written to a
public-facing wiki, and never be pasted into a chat channel that is
not the release-engineering-restricted private channel. The release
manager records every use of the token in the access log within
twenty-four hours of the release window closing.

The token's four segments encode, in order: the artefact's class
prefix (RT for "release token"), a four-digit serial that increments
strictly within a quarter, and two human-readable mnemonic segments
chosen at random from the canonical mnemonic set. The mnemonic
segments exist purely so an engineer reading the token aloud over a
voice channel cannot accidentally transpose digits — the mnemonic
words are unambiguous in spoken form even when the line is poor.

If a release manager believes the token has been compromised, the
escalation procedure is: page the release-engineering on-call, mark
the token revoked in the release log, generate a fresh token, file
the rotation as a high-priority operations ticket, and announce the
revocation in the release-engineering-restricted private channel.
Compromise is rare in practice — the most common cause of an
unscheduled rotation is a release manager pasting the token into the
wrong channel during a long incident response, and noticing within
seconds. Even in that case the rotation procedure is followed
strictly; we do not make exceptions for "obviously low-blast-radius"
mistakes.

## Chapter 2 — Build provenance

Every artefact that enters the production registry carries a build
provenance manifest. The manifest records: the source commit hash,
the build environment hash, the toolchain version, the timestamp,
the builder identity, and the set of dependencies pinned at build
time. The manifest is signed by the build system's offline signing
key and attached to the artefact at registry-upload time. A
production deploy that cannot show a valid manifest for the artefact
under deployment is refused at the deploy gate.

The signing key is held offline in the security team's hardware
security module. The key is rotated annually and on any compromise
event. Rotation is performed by the security team's principal
engineer with a witness; the procedure is documented in the security
runbook and is not reproduced here. The relevant point for release
engineering is that the verification key is published to the
production deploy gate's trust store and is checked on every deploy.

When a manifest fails verification, the deploy gate refuses the
deploy and pages the security on-call. The release manager does not
get to override the gate. This is a hard rule and has not been
violated since the gate was introduced.

## Chapter 3 — Canary rollouts

Canary rollouts are gated by automated SLO checks. The rollout runs
in three phases: a one-percent canary, a ten-percent canary, and a
full rollout. Each phase has a soak time during which the SLO
checker watches the canary's error budget; the rollout advances to
the next phase only when the soak time elapses without breaching the
budget. The default soak times are fifteen minutes, thirty minutes,
and sixty minutes for the three phases; individual services can
override these in their service descriptor when the team has a
specific reason to.

The SLO checker watches three signals: the error rate (HTTP 5xx as
a fraction of total requests), the latency distribution (p50 and
p95 against the service's declared SLO targets), and the saturation
indicators (queue depth, connection-pool utilisation, GC pause
distribution). A breach on any of the three triggers automatic
rollback. The release manager is paged on rollback so a human
investigates before the next attempt.

Rollback is itself a release with its own release token; the system
does not collapse rollback into a special case because that would
defeat the audit trail. The rollback artefact is the previous
production artefact, and its release token is the previous release's
token re-issued with a "ROLLBACK" suffix. This is the only case in
which a token segment carries human-readable suffixed metadata; the
suffix is part of the audit story, not the discriminator.

## Chapter 4 — Configuration management

Configuration is treated as code. Every configuration change goes
through the same review queue as a code change, and the same deploy
gate as a code deploy. There is no "I just need to flip a flag" path.
This rule was introduced after the 2024 incident in which a
production traffic-shaping flag was flipped without a review and
took down half the regional fleet for nineteen minutes. The
postmortem from that incident is the canonical reference for why the
rule exists.

Feature flags are a special case of configuration. A feature flag is
a runtime-evaluated boolean that gates a code path. Flags are
configured in the flag store, which is part of configuration
management. Adding a new flag is a code change; flipping a flag is a
configuration change. Both go through the queue.

The flag store has a two-tier evaluation model. The tier-one
evaluation runs at request entry and is cached for the lifetime of
the request; the tier-two evaluation runs at job-scheduling time and
is cached for the lifetime of the scheduled job. A flag is declared
as either tier-one or tier-two at flag-creation time; the tier
cannot be changed after the flag has been observed in production.
This rule exists because changing a flag's tier mid-flight would
introduce a difficult-to-reproduce class of bugs.

## Chapter 5 — Secrets management

Secrets are stored in the secret store. The secret store has a
hardware-rooted master key, a key-encryption-key per-tenant derived
from the master, and per-secret data-encryption keys. Each secret is
encrypted at rest under its own data key. The data keys are
themselves encrypted under the tenant's KEK and stored alongside
the encrypted secret. Only the secret-store service can decrypt;
clients fetch a decrypted secret over an authenticated channel and
the decryption is logged.

Secrets have a TTL and a rotation policy. The default TTL is thirty
days for ambient credentials and ninety days for high-value secrets;
high-value secrets are rotated proactively at half-life rather than
on expiry. Rotation is automated where the consuming service supports
it, and a manual handoff where it does not; either way the secret
store records the rotation event.

A secret that is past its TTL is refused at fetch time. This is a
hard rule and has caused outages — usually because a long-lived
service held a cached secret past its TTL and never re-fetched. The
fix is documented in the secret-store runbook: re-fetch secrets at a
cadence shorter than the TTL, and treat fetch errors as
authentication errors rather than retryable transients.

## Chapter 6 — Observability

Observability is built on three signals: structured logs, traces,
and metrics. The logs are written through the logging library and
are shipped to the log store; the traces are emitted through the
tracing library and are shipped to the trace store; the metrics are
emitted through the metrics library and are shipped to the metrics
store. The three stores are independent; there is no shared write
path. The reason for this is fault-isolation: a failure in one
observability backend does not cascade into the other two.

The log store retains data for ninety days. The trace store retains
data for thirty days. The metrics store retains data at one-minute
resolution for ninety days, at one-hour resolution for one year, and
at one-day resolution forever. The retention policies are tuned
against storage cost and the rate at which a postmortem investigation
needs to reach back; ninety days has historically been the upper
bound on a postmortem's interesting window.

Cross-signal correlation is achieved through a request id that flows
from the edge of the system to every downstream call. The request id
is a UUID minted at request entry and is propagated through HTTP
headers and message metadata. Every log line, span, and metric tag
carries the request id, so an investigator can reconstruct a single
request's behaviour across all three observability backends.

## Chapter 7 — Capacity planning

Capacity is planned on a quarterly cadence. Each service owner
submits a capacity request describing the service's expected
trajectory for the next two quarters: traffic forecast, latency SLO,
error-rate budget, and any anticipated spikes. The capacity team
aggregates the requests, projects against the underlying
infrastructure budget, and produces a capacity plan. The plan is
reviewed at the quarterly capacity review and signed off by the
director of platform engineering.

Capacity planning is intentionally a quarterly cadence. Annual
planning is too coarse; monthly planning is too fine and has the
unintended effect of flattening the team's strategic horizon to the
next month. Quarterly planning has worked for the team's size and
growth rate over multiple years; we revisit the cadence on an annual
basis as a check.

A service that exceeds its capacity allocation is paged. The page
goes to the service owner, not the capacity team — the assumption is
that a capacity overrun reflects a service-level problem (a hot
endpoint, an inefficient query, a runaway loop) before it reflects a
capacity-planning problem. The capacity team reviews overruns at the
weekly operations review and decides whether the overrun reflects a
genuine planning miss; if so, the capacity allocation is adjusted on
the spot.

## Chapter 8 — Incident response

The incident response procedure is described in the operations
handbook and is not reproduced in detail here. The release-
engineering-specific notes are: a rollout that triggers a rollback is
treated as a near-miss, not an incident, unless the rollback itself
fails to restore service within the SLO. A rollback that fails to
restore service is a severity-one incident automatically and is
treated as such regardless of the underlying cause. A near-miss is
reviewed at the next operations review and may produce action items;
an incident is reviewed at a postmortem.

The release manager is on call during the rollout window and remains
on call until either the full rollout completes or the rollback
completes. The release manager is not the same role as the on-call
engineer for the affected service; the two roles are kept distinct so
the release manager can focus on the rollout's progress and the
service on-call can focus on the service's behaviour.

## Chapter 9 — Documentation freshness

Every system in the release engineering domain has a runbook, and
every runbook has a freshness stamp. A runbook whose freshness stamp
is older than six months is considered stale. The release-engineering
on-call at the start of a rotation reviews the freshness state of
every runbook and either refreshes the stale runbooks or files a
ticket. The release-engineering tooling itself has a runbook in the
same store; that runbook is reviewed quarterly because the tooling
changes more rapidly than the systems it touches.

Architectural decisions are recorded in architectural decision
records (ADRs). An ADR is a short markdown document with a number, a
title, a status, a context section, a decision section, and a
consequences section. ADRs are written when a non-trivial choice is
made and are added to the ADR index in the documentation tree. The
ADRs are not retroactively edited; a superseding decision is recorded
as a new ADR that references the older one in its context section.

## Chapter 10 — Cross-team interfaces

The release engineering team interacts with the platform engineering
team, the security team, the SRE team, and the product engineering
teams. The cross-team interfaces are defined by the release
engineering charter, which is reviewed annually. The charter
specifies which of the team's outputs are consumed by which team and
how requests for changes to the team's outputs are routed.

In practice the cross-team interface that needs the most attention is
the interface with the security team. The release engineering team
is responsible for the integrity of the production registry and the
deploy gate; the security team is responsible for the trust store
and the signing keys. The two teams need to coordinate closely on
key rotations and on any change to the trust model. We coordinate
through a standing weekly meeting and an emergency escalation path.

## Chapter 11 — Tooling philosophy

The release engineering team owns several tools: the release
engineering tool, the deploy gate, the production registry, the
canary rollout controller, and the SLO checker. These tools are
built and maintained by the team in-house; we have considered
adopting external tooling on multiple occasions and have rejected
each on the basis that the integration cost exceeds the maintenance
cost. The decision is revisited annually; we do not assume the
historical answer is the right answer for the future.

The tools are written in the team's primary language and follow the
team's coding standards. Each tool has its own runbook, its own
postmortem history, and its own ADR set. The tools are not coupled
to one another at runtime — each is a self-contained service — but
they share a small library of common utilities. The shared library
is versioned and tools update on their own cadence; we do not
require synchronous upgrades.

## Chapter 12 — Onboarding

A new engineer on the release engineering team completes a six-week
onboarding programme. The programme covers: the team's tools, the
team's runbooks, the cross-team interfaces, the release procedure,
the rollback procedure, the canary controller, the SLO checker, and
the production registry. Each topic is paired with a hands-on
exercise; the exercises are designed so that the new engineer
performs the procedure in a non-production environment with a
simulated incident pattern.

The onboarding programme concludes with a practical exam: a
forty-five-minute simulated rollout in which the new engineer
executes a release end-to-end, observes a simulated SLO breach,
triggers the rollback procedure, and writes a postmortem. The exam
is scored against a rubric that mirrors the team's expectations for
a release manager. The new engineer is paired with a mentor for the
first three months after onboarding completes.

## Chapter 13 — Metrics that matter

The team tracks a small set of metrics: rollouts-per-quarter, mean
time to rollback, percentage of rollouts that completed without a
rollback, percentage of rollouts that completed within the planned
window, and the error rate of the deploy gate (which should be
zero). The metrics are reviewed at the operations review every week
and at the quarterly capacity review.

We avoid vanity metrics. The number of releases shipped is not a
goal metric; it is an output that depends on the product engineering
teams' shipping cadence. The team's contribution to throughput is
visible in the percentage of rollouts that complete without
intervention; that is the metric we optimise for.

## Chapter 14 — Failure modes we have seen

The most common failure modes the team has encountered are: a
mis-configured SLO check that triggered a false rollback; a
manifest-verification failure caused by a clock-skew in the build
environment; a flag-tier bug from before the tier-immutability rule
was introduced; a secret-rotation failure in a long-lived job that
held a cached secret past its TTL; and a release-token compromise
caused by a release manager pasting the token into a public chat
channel during a long incident response.

For each failure mode we have a documented postmortem and a
documented remediation. The remediations have been internalised into
the team's tooling and procedures; we do not rely on individual
engineers to remember the lessons.
